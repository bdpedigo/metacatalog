## ADDED Requirements

### Requirement: Standard response omits materialization fields
The standard `AssetResponse` schema returned to non-admin callers SHALL NOT include the `source` or `mat_version` fields.

#### Scenario: Non-admin lists assets
- **WHEN** a non-admin caller sends `GET /api/v1/assets/?datastack=example`
- **THEN** each asset in the response SHALL NOT contain `source` or `mat_version` keys

#### Scenario: Non-admin gets single asset
- **WHEN** a non-admin caller sends `GET /api/v1/assets/{id}`
- **THEN** the response SHALL NOT contain `source` or `mat_version` keys

### Requirement: Admin response includes materialization fields
An extended `AssetDetailResponse` schema SHALL include all fields from `AssetResponse` plus `source` and `mat_version`. This schema SHALL be returned to callers with `is_admin=True`.

#### Scenario: Admin lists assets
- **WHEN** an admin caller sends `GET /api/v1/assets/?datastack=example`
- **THEN** each asset in the response SHALL include `source` (string) and `mat_version` (integer or null) fields

#### Scenario: Admin gets single asset with materialization source
- **WHEN** an admin caller sends `GET /api/v1/assets/{id}` for an asset with `source="materialization"` and `mat_version=7`
- **THEN** the response SHALL include `"source": "materialization"` and `"mat_version": 7`

### Requirement: Query filter for mat_version restricted to admins
The `mat_version` query parameter on `GET /api/v1/assets/` SHALL only be accepted from admin callers. Non-admin callers providing `mat_version` as a query filter SHALL have it silently ignored.

#### Scenario: Non-admin filters by mat_version
- **WHEN** a non-admin caller sends `GET /api/v1/assets/?datastack=example&mat_version=5`
- **THEN** the system SHALL ignore the `mat_version` filter and return assets matching the other criteria only

#### Scenario: Admin filters by mat_version
- **WHEN** an admin caller sends `GET /api/v1/assets/?datastack=example&mat_version=5`
- **THEN** the system SHALL return only assets with `mat_version=5`

### Requirement: OpenAPI docs reflect schema split
The OpenAPI specification generated by FastAPI SHALL document both `AssetResponse` (standard) and `AssetDetailResponse` (admin) schemas. Endpoint response schema SHALL be documented as a union or as the superset with field-level descriptions noting admin visibility.

#### Scenario: OpenAPI schema includes both models
- **WHEN** a developer accesses `/docs` or `/openapi.json`
- **THEN** both `AssetResponse` and `AssetDetailResponse` schemas SHALL be present with descriptions indicating which fields are admin-only
